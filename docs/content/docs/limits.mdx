---
title: Limits and troubleshooting
description: Current tunnel limits and common setup failures.
---

## Current limits

- The public URL works only while `tt http` or `tt dev` is running.
- HTTP requests time out after 30 seconds.
- Request and response bodies are limited to 4 MB each.
- A tunnel accepts up to 32 concurrent public WebSocket connections.
- The local relay socket pool can contain 1 to 16 connections and defaults to 2.

## The local app is not reachable

Start the app before opening the tunnel. Confirm the port and, if needed, pass the interface it listens on:

```sh
tt http 5173 --host 127.0.0.1
```

The preflight request has a three-second timeout.

If the app is visibly running but the check still fails, compare its printed URL with the `Local` row. Fixed-port scripts need the same value passed to `tt dev --port`, and servers bound to another interface need `tt http --host`.

## No gateway is configured

Deploy one first:

```sh
tt deploy
```

For local gateway development, provide a domain, relay URL, and secret through options or environment variables instead.

## The Vercel CLI is missing or logged out

Install it and authenticate:

```sh
npm i -g vercel
vercel login
vercel whoami
```

Then retry `tt deploy`.

## A custom domain fails

The active Vercel account must own or have access to the domain. Review the Vercel CLI output, complete any required DNS verification, and run `tt deploy` again. Turbotunnel keeps the previous local config when deployment or verification fails.

## The config file is invalid

Fix the JSON in `~/.turbotunnel/config.json` or remove the file and deploy again. All supported fields are strings.

## Gateway verification fails

Open `https://<gateway-host>/_turbotunnel/status` and inspect the Vercel deployment logs. Verification retries after 1, 2, 4, and 8 seconds; each request has a 15-second timeout. The saved config changes only after verification passes.
