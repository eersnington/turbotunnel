---
title: tt http
description: Expose an already-running local HTTP or WebSocket app.
---

```sh
tt http <port> [options]
```

`tt http` exposes an already-running local app through the configured gateway. It checks the local target, opens a pool of relay WebSockets, and forwards traffic until the process stops.

Before connecting, terminal output shows the resolved public URL, local target, and relay count. The loading line becomes a stable readiness message after every relay connects. Redirected output and continuous integration (CI) runs use plain, non-animated milestones.

## Arguments and options

| Input               | Description                                                                            | Default                        |
| ------------------- | -------------------------------------------------------------------------------------- | ------------------------------ |
| `<port>`            | Local TCP port, from 1 to 65535.                                                       | Required                       |
| `--slug <slug>`     | Public tunnel slug. Use lowercase letters, digits, and hyphens; maximum 63 characters. | Saved slug or a random slug    |
| `--host <host>`     | Local app host.                                                                        | `localhost`                    |
| `--pool <count>`    | Number of local relay sockets, from 1 to 16.                                           | `2`                            |
| `--domain <domain>` | Base tunnel domain or `{slug}` host pattern for this run.                              | Saved domain                   |
| `--secret <secret>` | Relay secret for local gateway development.                                            | Saved secret                   |
| `--relay-url <url>` | Explicit `http`, `https`, `ws`, or `wss` relay origin.                                 | Derived from the tunnel domain |

## Common tunnel configurations

Expose a Vite app:

```sh
tt http 5173
```

Connect to an app listening on another interface:

```sh
tt http 3000 --host 127.0.0.1
```

Choose a public slug after deploying a wildcard custom domain:

```sh
tt deploy --domain tunnel.example.com
tt http 5173 --slug demo
```

This produces `https://demo.tunnel.example.com/`. The default `<slug>-turbotunnel.vercel.app` domain supports only the slug in the deployed Vercel project name.

Increase the relay pool to add direct HTTP capacity and independent relay connections:

```sh
tt http 5173 --pool 4
```

## Local gateway development

Point the client at a gateway running locally:

```sh
tt http 5173 \
  --domain localhost:3002 \
  --relay-url ws://127.0.0.1:3002 \
  --secret dev_secret
```

When `--relay-url` is set, Turbotunnel still uses the configured domain to build the public host and sends that host to the relay.

## Startup checks

Turbotunnel sends `HEAD /` to `http://<host>:<port>/` before connecting. Any HTTP response proves the app is reachable. The check times out after three seconds.

If no gateway settings are available from options, environment variables, or the saved config, run `tt deploy` first.
