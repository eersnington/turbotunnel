This page describes repository projects, saved gateway settings, and the production variables managed by `tt deploy`.

## Project configuration

Place `turbotunnel.json` in a repository directory. Turbotunnel searches upward from your current directory and uses the nearest file. Named projects contain tunnel settings only:

```json
{
  "access": { "type": "public" },
  "projects": {
    "dashboard": {
      "port": 3000,
      "slug": "dashboard"
    },
    "docs": {
      "port": 5173,
      "domain": "docs.eers.dev"
    }
  }
}
```

Run `tt dev dashboard` to select a named project. With one configured project, `tt dev` selects it automatically. With multiple projects, an interactive terminal presents a selector. A single-project configuration can put `port`, `slug`, `domain`, and `access` at the top level.

`slug: "dashboard"` first requests `dashboard-turbotunnel.vercel.app`. If another Vercel project owns that name, Turbotunnel assigns and saves `tt<random>-dashboard.vercel.app`. `domain` selects one exact custom hostname. You cannot combine `domain` with `slug`.

Supported access policies are `public`, `password`, and `ip` with an `allow` array. Password access generates a fresh `tt_<nanoid>` password for each session. Pass `--password <value>` to use a custom value. Turbotunnel never writes passwords to configuration.

### Configuration fields

| Field      | Type                                                          | Default                 | Description                                                            |
| ---------- | ------------------------------------------------------------- | ----------------------- | ---------------------------------------------------------------------- |
| `port`     | Integer from 1 to 65535                                       | `--port`                | Local HTTP port targeted by the tunnel.                                |
| `slug`     | Lowercase letters, digits, and hyphens; maximum 63 characters | Generated or saved slug | Requests a managed `<slug>-turbotunnel.vercel.app` hostname.           |
| `domain`   | Hostname string                                               | None                    | Uses one exact custom public hostname. Cannot be combined with `slug`. |
| `access`   | Access policy object                                          | `public`                | Controls who can reach the public tunnel.                              |
| `projects` | Object of named project fields                                | None                    | Defines multiple selectable tunnel projects.                           |

Unknown fields are rejected. Project configuration does not define commands, working directories, environment variables, or package-manager behavior.

### Access policies

Public access requires no authentication:

```json
{ "access": { "type": "public" } }
```

Password access generates and displays a new password when the tunnel starts:

```json
{ "access": { "type": "password" } }
```

IP access requires at least one IPv4 or IPv6 address or CIDR:

```json
{
  "access": {
    "type": "ip",
    "allow": ["203.0.113.10", "2001:db8::/32"]
  }
}
```

In a named-project configuration, top-level `access` is the default for every project. A project's own `access` policy overrides it:

```json
{
  "access": { "type": "password" },
  "projects": {
    "dashboard": { "port": 3000 },
    "docs": {
      "port": 5173,
      "access": { "type": "public" }
    }
  }
}
```

### Single-project configuration

When a repository needs one tunnel, omit `projects` and put the project fields at the top level:

```json
{
  "port": 3000,
  "domain": "dashboard.example.com",
  "access": { "type": "password" }
}
```

## Saved configuration

`tt deploy` writes `~/.turbotunnel/config.json` after it verifies the gateway:

```json
{
  "project": "ttabc123-turbotunnel",
  "slug": "ttabc123",
  "relayDomain": "{slug}-turbotunnel.vercel.app",
  "queueRegion": "iad1",
  "relaySecret": "ttsec_example_redacted"
}
```

The relay secret authenticates local relay connections, access cookies, and requests to the tunnel-list endpoint. The file is written with owner-only permissions. Do not commit or share it.

`tt deploy` also keeps generated deployment files in `~/.turbotunnel/relay`. It preserves that directory's `.vercel` project metadata between deployments.

`tt http` resolves configuration in this order:

1. Command-line options
2. Repository project configuration
3. Saved gateway configuration
4. Generated defaults

## Vercel environment variables

`tt deploy` sets these variables in the project's Production environment:

| Variable                   | Value                                       |
| -------------------------- | ------------------------------------------- |
| `TURBOTUNNEL_BASE_DOMAIN`  | The configured base domain or host pattern. |
| `TURBOTUNNEL_RELAY_SECRET` | The generated or reused relay secret.       |
| `TURBOTUNNEL_QUEUE_REGION` | The configured queue region.                |

## Local gateway development

The local gateway listens on port `3002` with an in-memory broker. It defaults to domain `localhost`, relay secret `dev_secret`, and Queue region `iad1`. Do not use the development secret for a public deployment.
